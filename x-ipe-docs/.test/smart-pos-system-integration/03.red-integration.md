# [FSN] SMART POS <-> RED Integration Design

> Integration design for SMART POS and RED (RFID EPC management system) for FSN division, covering EPC-based receiving, stock taking, and sales transaction data flows.

## Integration Overview

RED is the RFID EPC (Electronic Product Code) management system integrated with SMART POS for FSN division. It handles EPC data for product receiving, stock taking, and sold marking after sales. Integration is API-based and mediated through **Apollo CN** and **IPAAS CN**.

### Key Facts
- **Division:** FSN only
- **Integration Type:** API
- **Gateway:** Apollo CN, IPAAS CN
- **Performance Requirement:** Per API < 1.5 sec
- **Security:** Chanel Intranet access only

## Business Flow Impact

| Domain | Function | Integration Type | Caller | Callee |
|--------|----------|-----------------|--------|--------|
| Inventory Management | Push Receiving Order (EPC) | API | RED | SMART POS |
| Sales Management | Sync TX (EPC Sold Mark) | API | SMART POS | RED |
| Inventory Management | Push Stock Taking (EPC) | API | RED | SMART POS |

### Downstream Functions
- EPC validation during sales
- EPC sold mark after sales
- Stock taking via EPC

## Integration Findings & Assumptions

| Category | Finding/Assumption |
|----------|-------------------|
| Enterprise Direction | Single source of truth for EPC is from RED |
| Enterprise Direction | Sync TX uses common pattern: forward to Apollo APAC for distribution |
| Target System | RED has no process flow change for SMART POS |
| Design Reference | Solution referenced from AS-IS TTPOS lessons learned |
| Integration Gateway | Apollo CN required for SMART POS integration |
| Integration Gateway | Apollo APAC required for TX distribution pattern |

## Architecture Comparison (AS-IS vs TO-BE)

- **Push Receiving Order:** AS-IS TTPOS and TO-BE SMART POS practices are the same
- **Sync TX:** AS-IS TTPOS and TO-BE SMART POS practices are the same
- **Push Stock Taking:** AS-IS TTPOS relied on DataLab; TO-BE SMART POS leverages in-app report (business needs driven)

## Low-Level Design — Flow Deep Dives

### 1. Push Receive Order
- **ORLI Code** used as boutique code for RED
- **IPAAS CN** used when RED sends EPC data to China
- **Parallel Run:** Receiving order distributed based on ORLI code to either TTPOS or SMART POS
- **Cut-Over Changes:** Apollo no longer distributes to TTPOS based on ORLI code; ORLI-BTQ mapping sync removed

**Key Design Specs:**
| Flow | Protocol | Trigger | Method |
|------|----------|---------|--------|
| Sync ORLI Code & BTQ Mapping | File | Event (Boutique Code CUD) | Instance |
| Push Receiving Order | API | Event | Instance |

### 2. Sync TX
- TX data synced from SMART POS to RED via Apollo APAC distribution pattern
- Parallel run and cut-over designs documented

### 3. Push Stock Taking
- Stock taking data pushed from RED to SMART POS
- Leverages SMART POS in-app report capability (different from TTPOS which used DataLab)
