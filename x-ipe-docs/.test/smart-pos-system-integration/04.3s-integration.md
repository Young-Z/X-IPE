# [FSN] SMART POS <-> 3S Integration Design

> Integration design for SMART POS and 3S (EPC/SN validation service) for FSN division, covering EPC and Serial Number validation during sales.

## Integration Overview

3S provides EPC (Electronic Product Code) and SN (Serial Number) validation services for the FSN division of SMART POS. The integration enables real-time validation of product authenticity during sales transactions.

### Key Facts
- **Division:** FSN only
- **Integration Type:** API (3S V2 API)
- **Gateways:** Apollo CN, Global APIM
- **Performance Requirement:** Per API < 1.5 sec
- **Monitoring:** Dynatrace for long-term monitoring

## Business Flow Impact

| Domain | Function | Type | Caller | Callee |
|--------|----------|------|--------|--------|
| Sales Management | Validate EPC & SN Status via EPC Code | API | SMART POS | 3S |

**Downstream:** SN validation during sales

## Integration Findings & Assumptions

| Category | Finding/Assumption |
|----------|-------------------|
| Target System | 3S V2 API proposed to be used |
| Target System | 3S calls SN360 for SN validation; SMART POS depends on 3S for SN validation |
| Design Reference | Referenced from AS-IS TTPOS lessons learned |
| Design Reference | If 3S unavailable, keep 2S route as TTPOS fallback |
| Design Reference | Referenced from Star POS lessons learned |
| Integration Gateway | Apollo CN required for SMART POS integration |
| Integration Gateway | Global APIM required for 3S integration |

## Architecture Comparison

- AS-IS: Touchpoint not actually using this route; TO-BE used by ICON and SMART POS
- AS-IS: TTPOS integrated with 3S V1 API; TO-BE uses V2
- AS-IS: TTPOS relied on Apollo APAC (handshake only) for 3S V1 API; TO-BE uses V2

## Low-Level Design

### Flow: Validate EPC & SN Status via EPC Code
- **Protocol:** API (3S V2)
- **Integration path:** SMART POS → Apollo CN → Global APIM → 3S
- 3S internally calls SN360 for SN validation
