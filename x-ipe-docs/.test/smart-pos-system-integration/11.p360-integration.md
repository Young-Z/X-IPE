# [WFJ] SMART POS <-> P360 Integration Design

> Integration design for SMART POS and P360 for WFJ division (also FSN/FBP), providing serial number (SN) validation during sales.

## Integration Overview

P360 provides serial number validation for watches/jewelry. SMART POS calls P360 to validate SN during sales transactions. The single source of truth for SN validation is SN360.

### Key Facts
- **Divisions:** FSN, WFJ, FBP (all three)
- **Integration Type:** API
- **Gateways:** Apollo CN → Global APIM → P360 (Apollo APAC can be omitted since P360 is a global system)
- **Security:** ADFS Clasp for authorization
- **Performance:** Per API < 1.5 sec

## Business Flow Impact

| Domain | Function | Type | Request | Response |
|--------|----------|------|---------|----------|
| Sales Management | SN Validation | API | SMART POS | P360 |

## Integration Findings & Assumptions

| Category | Finding/Assumption |
|----------|-------------------|
| Enterprise Direction | Single source of truth for SN validation should be SN360 |
| Target System | P360 process flow assumed unchanged |
| Design Reference | Referenced from AS-IS TTPOS and APAC Star POS lessons learned |
| Integration Gateway | Global APIM required for P360; Apollo CN for SMART POS |
| Integration Gateway | Apollo APAC can be omitted (P360 is global — not necessary to route APAC→CN) |
| Security | ADFS Clasp for authorization |

## Low-Level Design

### Flow: SN Validation
- **Protocol:** API
- **Path:** SMART POS → Apollo CN → Global APIM → P360
- **Parallel Run:** TTPOS keeps AS-IS; SMART POS uses same design for parallel run and cut-over
