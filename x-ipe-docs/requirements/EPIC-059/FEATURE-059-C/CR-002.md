# Change Request: CR-002 — DAO-Owned Active Tracking Analysis Flow

> Created: 2026-04-27
> Status: Documented
> Classification: Modification to FEATURE-059-C
> Related Feature: FEATURE-059-C

## Request Details

**Requestor:** Human product owner
**Date Received:** 2026-04-27
**Priority:** High

## Description

Update the CR-001 active tracking flow so the Knowledge Librarian DAO routes behavior-learning requests to the mimic skill, while the mimic skill owns active polling and consolidation:

1. Active behavior learning must be routed through `x-ipe-assistant-knowledge-librarian-DAO`, which calls `x-ipe-knowledge-mimic-web-behavior-tracker`.
2. `x-ipe-knowledge-mimic-web-behavior-tracker.start_active_tracking` must own active polling until the user clicks the **Analysis** button in the tracker toolbar.
3. When the user clicks **Analysis**, `start_active_tracking` must detect it through `poll_tick`, stop tracking, consolidate collected information, and pass the result back to the DAO for downstream knowledge-pipeline use.

## Business Justification

CR-001 restored parity with the retired behavior tracker, but left the active loop owned by a generic orchestrating agent and terminated by `stop_tracking` / tracker-loss semantics. That does not match the intended knowledge-learning flow: the Knowledge Librarian DAO should identify/delegate mimic tasks, the mimic skill should own active polling/consolidation, and the human should decide when enough behavior has been captured by clicking **Analysis**.

### Conflict Analysis

**Conflicts Found:** Yes

| # | Type | Affected Feature | Severity | Classification | Description |
|---|------|------------------|----------|----------------|-------------|
| 1 | design | FEATURE-059-C | High | expected | CR-001 says the generic agent owns polling and stops on `stop_tracking` / `TRACKER_LOST`; CR-002 requires mimic-owned polling inside `start_active_tracking` and Analysis-button termination. |
| 2 | specification | FEATURE-059-C | High | expected | AC-059C-17a currently says the agent loop runs until `stop_tracking`; this must be superseded for DAO learning flows. |
| 3 | integration | FEATURE-059-E / Librarian DAO | Medium | expected | Librarian DAO currently discovers/runs knowledge skills but does not explicitly classify behavior-learning requests as mimic tasks or consume mimic's final Analysis handoff. |
| 4 | implementation risk | Mimic tracker toolbar | Medium | unexpected | The Analysis flag can be reset by `getAnalysisFlag()` and can be lost during reinjection/navigation unless made sticky or persisted. |

**Unexpected Conflict Resolution:**
- Make Analysis request detection sticky for active sessions until the DAO acknowledges/consumes the handoff result.
- Document that active tracking remains a single mimic-owned loop per `start_active_tracking` invocation; concurrent pollers are out of scope.

**Mitigation Strategy:**
- Supersede only CR-001’s loop owner and stop condition. Keep CR-001’s useful behaviors: 5s polling, auto-reinject, auto-screenshot on new events, and single accumulating `track-list.json`.
- Preserve standalone degraded-mode behavior where possible: direct callers may still call mimic operations, but the supported "learn with mimic" path goes through the Librarian DAO.

## Proposed Approach

1. Add a Librarian DAO path for "learn from website/application behavior" that delegates to the mimic skill.
2. `start_active_tracking` owns the polling loop every `polling_interval_s` seconds and internally invokes `poll_tick` until `poll_tick` returns `analysis_requested: true`.
3. `poll_tick` always merges events, handles URL-change reinjection, and handles screenshots as before. When Analysis is requested, it returns the signal and latest event state to `start_active_tracking`.
4. `start_active_tracking` handles Analysis completion:
   - flush/merge latest events
   - stop tracking
   - run post-processing / consolidate collected information
   - return `observation_summary` / `observations` to the DAO
   - mark the handoff as consumed so the toolbar can reset Analysis UI
5. DAO receives the handoff payload and uses it as knowledge input for downstream construction/extraction/ontology steps.
