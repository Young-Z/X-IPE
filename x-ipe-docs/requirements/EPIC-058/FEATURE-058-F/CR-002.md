## Description

Synthesize a search-context graph when BFS search returns matches across multiple disconnected graph files. Currently, searching a term like "lion" may return matched entities from different ontology graph files (e.g., `05.cass-integration.md`). If those matched entities have no pre-existing `relate` events connecting them, the subgraph returned by `_bfs_subgraph()` contains only disconnected nodes with no edges — resulting in a scattered, unconnected visualization on the canvas.

**Current behavior:** BFS search returns matched entities + their BFS-expanded neighborhoods. If matched entities span multiple graph files with no inter-graph relations, the subgraph nodes appear as disconnected islands on the canvas. The user sees isolated highlighted nodes with no visual indication of WHY they are related.

**Desired behavior:** When search results contain disconnected components, a virtual "search hub" node is synthesized, labeled with the search query keyword. Virtual edges (star topology) connect the hub to each direct-match entity. This hub + star edges are injected into the subgraph response and rendered on the canvas, providing visual context: "these entities are all related through the search term."

## Business Justification

The current BFS search finds the right nodes but fails to communicate their relationship when they come from different graph files. A user searching "lion" might see 5 highlighted nodes scattered across the canvas with no connecting edges — giving the impression the search found unrelated things. The virtual search hub node makes the implicit relationship explicit: all these entities matched "lion" and are contextually related through that search term. This dramatically improves the search-as-exploration UX.

## Proposed Approach

1. **Backend (`search.py`):** After `_bfs_subgraph()` returns, detect disconnected components among direct-match nodes. If >1 disconnected component exists, create a virtual hub node (`__search_hub__{query}`) and virtual edges from hub to each direct-match entity. Add to `subgraph.nodes` and `subgraph.edges`.

2. **Frontend (`ontology-graph-canvas.js`):** Enhance `highlightSubgraph()` to:
   - Detect virtual node IDs (prefix `__search_hub__`)
   - `cy.add()` the virtual hub node with position at centroid of matched nodes
   - `cy.add()` virtual edges from hub to each matched node
   - Apply special CSS class `search-hub` for distinct styling (dashed border, search icon)
   - On `clearHighlight()`, remove all virtual elements

3. **Frontend (`ontology-graph-socket.js`):** Pass virtual node metadata (label, node_type) alongside IDs so the canvas can render them properly.

4. **Data format:** Enrich `subgraph` to include node metadata for virtual nodes:
   ```json
   {
     "nodes": ["entity_1", "entity_2", "__search_hub__lion"],
     "edges": [["__search_hub__lion", "search_match", "entity_1"], ["__search_hub__lion", "search_match", "entity_2"]],
     "virtual_nodes": [{"id": "__search_hub__lion", "label": "lion", "node_type": "search_hub"}]
   }
   ```
