## Description

Rename `process_preference.auto_proceed` to `process_preference.interaction_mode` across the entire X-IPE system, replacing terse enum values with semantically meaningful keys that AI agents can parse without lookup tables.

**Key rename:**
- Config key: `auto_proceed` → `interaction_mode`

**Enum value renames:**
| Old Value | New Value | Meaning |
|-----------|-----------|---------|
| `auto` | `dao-represent-human-to-interact` | DAO represents the human at ALL touchpoints |
| `stop_for_question` | `dao-represent-human-to-interact-for-questions-in-skill` | DAO represents human for inner-skill questions only |
| `manual` | `interact-with-human` | Human handles all touchpoints directly |

**UI changes:**
- Add "Interaction Mode" label above/beside the dropdown
- Replace display labels: "Manual" → "👤 Human Direct", "Auto" → "🤖 DAO Represents Human", "Stop for Q" → "🤖⚡ DAO Inner-Skill Only"

**CLI flag changes:**
- `--execute@keep-running-forever` → `--interaction@dao-represent-human-to-interact`
- `--execute@keep-running-forever-stop-only-on-question` → `--interaction@dao-represent-human-to-interact-for-questions-in-skill`

**Backward compatibility:** Backend MUST accept both old and new values, auto-migrating old values on read.

## Business Justification

The current `auto_proceed` naming predates the DAO skill architecture (EPIC-047). Now that `x-ipe-assistant-user-representative-Engineer` is the actual mechanism behind "auto" mode, the terminology is misleading:
1. "Auto" suggests generic automation — but the real behavior is DAO *representing the human*
2. AI agents parse these keys at every decision point — semantic keys prevent misinterpretation without lookup tables
3. Design principle: **meaningful is over simplicity** for AI-consumed configuration
