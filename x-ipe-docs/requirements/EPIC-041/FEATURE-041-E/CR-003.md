## Description

Update the deliverable tag system to support **multiple output files per tag**. Currently, each deliverable tag (`$output:tag-name`) maps to a single file path string. This CR introduces array-value support so that a tag can hold multiple file paths.

**Concrete example:** The `compose_idea` action produces:
- `new idea.md` (the idea document)
- `ńö¤µłÉµĖ®µĢģĶĆīń¤źµ¢░ÕøŠńēć.png` (uploaded reference image)
- `ńö¤µłÉĶāīµÖ»ÕøŠ.png` (uploaded reference image)

Currently, only `new idea.md` appears in the deliverables. The two uploaded PNGs are in the ideas folder but not individually listed. After this CR, all three files should appear as deliverable cards.

**Tag rename:** `$output:raw-idea` (singular) ŌåÆ `$output:raw-ideas` (plural) to indicate multi-value support.

## Business Justification

- Users upload reference images alongside idea documents during compose_idea
- These uploaded files are invisible in the deliverables section, making them hard to discover
- Downstream actions (refine_idea) should be aware of ALL inputs from compose_idea
- Improves workflow transparency and traceability

**Mitigation Strategy:**
- **Backward compatibility:** Support both string and array values in storage/resolution. If value is a string, treat as single-element array internally.
- **Template resolution:** For prompt templates (`$output:raw-ideas$`), resolve to the **first element** (primary file). Skills receive the primary file path as before.
- **Deliverable rendering:** For the UI, expand array into individual cards (one per file).
- **Schema versioning:** Bump to v4.0 when array values are first stored.

## Proposed Approach

1. **Template changes:** Rename tag in `workflow-template.json` and `copilot-prompt.json`
2. **Backend (storage):** Accept both `str` and `list[str]` values in deliverable dicts
3. **Backend (resolution):** When resolving for UI, expand arrays into individual entries. When resolving for templates/candidates, use first element.
4. **Frontend (rendering):** Render each array element as its own deliverable card
5. **Frontend (templates):** `_resolveTemplate` uses first element of array for substitution
6. **Validation:** Accept both string and array values; validate each array element
7. **Migration:** Duck-type existing instances ŌĆö string values treated as single-element array
